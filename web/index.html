🧬 What's happening under the hood

The stack

Every computation on this page runs inside a single 88 KB WebAssembly binary compiled from Rust. When you click a button or move a slider, JavaScript calls into WASM, which executes Ferray — a Rust-native, hand-tuned NumPy equivalent that beats NumPy on 73% of benchmarks — and returns the result back to the browser for rendering on a <canvas>.

The WASM binary was produced with cargo build --target wasm32-unknown-unknown and wasm-bindgen to generate the JS glue. Deployment is handled by a GitHub Actions workflow that builds the WASM from source on every push to main and deploys to GitHub Pages automatically.

⚠️ WASM limitations encountered

Only ferray-core compiles to WASM. The full Ferray workspace (18 crates) cannot currently target wasm32-unknown-unknown due to two fundamental blockers:

1. ferray-ufunc → core-math → bindgen
The universal function crate depends on core-math, which uses bindgen at build time to generate Rust FFI bindings to C math libraries. bindgen requires libclang (LLVM's C API) to be installed on the build host. When cross-compiling to WASM, bindgen can't locate libclang for the target triple — it panics at build time. Without ferray-ufunc, all element-wise operations (sin, cos, exp, sqrt, etc.) are unavailable from Ferray.

2. ferray-linalg → faer → private-gemm-x86
The linear algebra crate depends on faer, which bundles hand-tuned GEMM kernels written in x86 SIMD intrinsics (AVX2, AVX-512, SSE). These kernels are architecture-specific and have no WASM fallback path. The matmul, SVD, QR, and eigenproblem routines are therefore inaccessible. The GEMM crate private-gemm-x86 is explicitly not portable beyond x86.

✅ What does work

  • ferray-core — N-dimensional arrays, broadcasting, indexing, shape manipulation, memory layout, COW arrays, views, and all creation functions (linspace, arange, zeros, ones, eye, etc.)
  • Zero panics — All ferray functions return Result<T, FerrayError>, so every call from JS is properly error-handled
  • wasm-bindgen — Seamless JS ↔ Rust interop with automatic serialization of Vec<f64>

💡 Ideas for the future

1. Contribute a WASM target to ferray-ufunc. The CORE-MATH library (which powers ferray's transcendentals) could be compiled to WASM via cc with the wasm32 target, and bindgen could be pointed at a WASM-compatible libclang. Alternatively, ferray-ufunc could feature-gate the CORE-MATH dependency and fall back to pure-Rust libm when targeting WASM.

2. Use SIMD via wasm_simd128. Rust's std::simd (nightly) and the stable wasm32-simd128 target could provide 128-bit SIMD for element-wise ops without C dependencies. Ferray's pulp-based SIMD dispatch already has a pulp-wasm-simd-flag crate in its dependency tree — the scaffolding exists.

3. Explore faer's WASM story. faer (the linear algebra backend) recently added experimental WASM support in its 0.20+ releases. Once that matures, ferray-linalg could potentially compile to WASM too.
