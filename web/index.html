🧬 What's happening under the hood

The stack

Every computation on this page runs inside a single 88 KB WebAssembly binary compiled from Rust. When you click a button or move a slider, JavaScript calls into WASM, which executes Ferray — a Rust-native, hand-tuned NumPy equivalent that beats NumPy on 73% of benchmarks — and returns the result back to the browser for rendering on a <canvas>.

The WASM binary was produced with cargo build --target wasm32-unknown-unknown and wasm-bindgen to generate the JS glue. Deployment is handled by a GitHub Actions workflow that builds the WASM from source on every push to main and deploys to GitHub Pages automatically.

⚠️ WASM limitations — a deep dive into the source

We audited the Ferray source code to understand exactly what blocks WASM compilation — and whether the developers already anticipated this target. The answer surprised us.

The architecture is already WASM-ready

Ferray uses a multi-backend dispatch architecture at every level of the stack. Every hot path checks the target architecture at compile time and falls through to a portable pure-Rust implementation when no hardware SIMD is available:

// ferray-linalg/src/gemm/mod.rs — actual source
#[cfg(target_arch = "x86_64")] → AVX2/AVX-512 hand-tuned kernels
#[cfg(target_arch = "aarch64")] → NEON kernels
#[cfg(not(x86_64 or aarch64))] → faer_fallback.rs (pure Rust matmul)
    // Comment: "RISC-V, WASM, big-endian PowerPC, etc."

Ferray's dual math strategy

The ufunc crate uses a split approach for math accuracy. Each function picks one of two backends — and half of them already work on WASM:

Functions Backend Accuracy WASM?
sin, cos, tan Float::sin (libm) ~1 ULP ✅ pure Rust
arcsin, arccos, arctan, sinh, cosh, tanh, exp, log, expm1, log1p, … CrMath (CORE-MATH) <0.5 ULP ❌ C FFI via bindgen
sin_fast, cos_fast, tan_fast Cody-Waite + Cephes polynomials ≤1 ULP ✅ pure Rust
exp_fast Even/Odd Remez decomposition ≤1 ULP ✅ pure Rust
add, multiply, sqrt, abs, square, … pulp SIMD / scalar fallback Exact ✅ pure Rust

The real blocker: just one crate, one file

After auditing the entire Ferray source tree, the WASM problem reduces to exactly one thing:

ferray-ufunc → core-math → bindgen → libclang

The core-math crate is a thin wrapper around core-math-sys, which uses bindgen to generate Rust FFI bindings to INRIA's CORE-MATH C library at build time. bindgen requires libclang (LLVM's C API), which can't resolve for the wasm32-unknown-unknown target. There is no pre-generated bindings file, no wasm feature, and no pure-Rust fallback in the core-math crate.

This cascades to everything: ferray-stats depends on ferray-ufunc. ferray-fft depends on it. Even ferray-linalg lists it as a dependency (though we found it never imports anything from it — it's a dead dep). So every downstream crate is blocked by this single C FFI boundary.

Correction from earlier analysis: the ferray-linalg → private-gemm-x86 concern was a red herring. The linalg GEMM dispatch already has a pure-Rust faer fallback for non-x86/non-aarch64 targets, and WASM is explicitly mentioned in the source comments alongside RISC-V and PowerPC. Ferray-linalg is architected for WASM — it just can't compile because of the transitive ferray-ufunc dependency.

✅ What does work

  • ferray-core — N-dimensional arrays, broadcasting, indexing, shape manipulation, memory layout, COW arrays, views, and all creation functions (linspace, arange, zeros, ones, eye, etc.)
  • Zero panics — All ferray functions return Result<T, FerrayError>, so every call from JS is properly error-handled
  • wasm-bindgen — Seamless JS ↔ Rust interop with automatic serialization of Vec<f64>
  • ferray-linalg (theoretically) — The GEMM dispatch, SVD, QR, decompositions, and solvers all route to pure-Rust faer when not on x86/aarch64. The crate just can't compile because Cargo resolves its dead ferray-ufunc dependency.

💡 How to fix it — a ~2 hour patch

The fix is surgically small. Every CORE-MATH function called in ferray-ufunc/src/cr_math.rs has an equivalent in the pure-Rust libm crate. The entire diff would be:

  1. Add libm = { version = "0.2", optional = true } and a wasm = ["dep:libm"] feature to ferray-ufunc/Cargo.toml
  2. In cr_math.rs: cfg-gate the existing impl CrMath for f64/f32 blocks behind #[cfg(not(feature = "wasm"))]
  3. Add #[cfg(feature = "wasm")] versions that call libm::asin() / libm::acos() / libm::exp() / etc. instead of core_math::asin()
  4. In ferray-linalg/Cargo.toml: make ferray-ufunc optional or remove the unused dependency

The trade-off: on WASM you'd get ~1-2 ULP accuracy (libm) instead of <0.5 ULP (CORE-MATH) for arcsin/arccos/arctan/sinh/cosh/exp/log — the same accuracy NumPy ships with. Everything else (all arithmetic, reductions, linalg, FFT) would work identically.

Why this matters: the ferray developers explicitly designed for WASM. The faer_fallback.rs comments call it out by name. The multi-arch dispatch already handles x86, aarch64, and "everything else". The _fast math variants are already pure Rust. The only missing piece is one #[cfg] gate in cr_math.rs. The architecture was ready — they just never wired the last switch.
