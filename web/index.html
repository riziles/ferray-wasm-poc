🧬 What's happening under the hood

The stack

Every computation on this page runs inside a single ~213 KB WebAssembly binary compiled from Rust. When you click a button or move a slider, JavaScript calls into WASM, which executes Ferray — a Rust-native, hand-tuned NumPy equivalent that beats NumPy on 73% of benchmarks — and returns the result back to the browser for rendering on a <canvas>.

The WASM binary bundles ferray-core (N-dimensional arrays), ferray-ufunc (SIMD-accelerated sin/cos/exp/log/…), and ferray-stats (min/max/mean/median/std/var/…). It was produced with cargo build --target wasm32-unknown-unknown and wasm-bindgen. Deployment is handled by a GitHub Actions workflow that builds from source on every push and deploys to GitHub Pages automatically.

✅ WASM — SOLVED

The original blocker was a single dependency chain that prevented all of Ferray beyond ferray-core from compiling to WebAssembly:

ferray-ufunc → core-math → core-math-sys → bindgen → libclang

The core-math crate (INRIA CORE-MATH) wraps a C library via bindgen, which requires libclang at build time. libclang can't resolve for wasm32-unknown-unknown, transitively blocking ferray-stats, ferray-linalg, ferray-random, and ferray-polynomial.

This is now fixed. We shipped a 3-commit patch to riziles/ferray-riz-dev that swaps core-math for the pure-Rust libm crate on wasm32 using target-conditional dependencies — no feature flags, no build.rs. Just cargo build --target wasm32-unknown-unknown.

How the fix works

Cargo's target-conditional dependencies do all the heavy lifting:

// ferray-ufunc/Cargo.toml — native gets CORE-MATH, WASM gets libm
[target.'cfg(not(target_arch = "wasm32"))'.dependencies]
core-math = { workspace = true } // <0.5 ULP on native

[target.'cfg(target_arch = "wasm32")'.dependencies]
libm = { workspace = true } // ~1-2 ULP on WASM (same as NumPy!)

Ferray's multi-backend architecture

Ferray was architected for WASM from the start — every hot path uses compile-time target detection with pure-Rust fallbacks:

// ferray-linalg/src/gemm/mod.rs — actual source
#[cfg(target_arch = "x86_64")] → AVX2/AVX-512 hand-tuned kernels
#[cfg(target_arch = "aarch64")] → NEON kernels
#[cfg(not(x86_64 or aarch64))] → faer_fallback.rs (pure Rust matmul)
    // Comment: "RISC-V, WASM, big-endian PowerPC, etc."

// ferray-ufunc/src/cr_math.rs — the fix
#[cfg(not(target_arch = "wasm32"))] → CORE-MATH (<0.5 ULP, via FFI)
#[cfg(target_arch = "wasm32")] → libm (~1-2 ULP, pure Rust)

The trade-off: on WASM you get ~1-2 ULP accuracy (libm) instead of <0.5 ULP (CORE-MATH) for transcendentals — the exact same accuracy NumPy ships with. Everything else (arithmetic, reductions, linalg, FFT) works identically.
